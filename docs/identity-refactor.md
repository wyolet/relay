# Identity refactor: `id` PK + `displayName` + `slug`

**Status:** planned, not started. Single PR, big.
**Owner:** next session pick this up.
**Why:** today `metadata.name` is both the PK and the user-typed label. Renaming changes the PK, which forces cascading reference updates and a dedicated `/rename` endpoint per kind. The frontend's UX wants to "just edit the name" — and operators want to fix typos without breaking every Policy that referenced the old name.

This refactor splits identity into three concerns:

| Field         | Role                                      | Mutability                | Generated by |
|---------------|-------------------------------------------|---------------------------|--------------|
| `id`          | Primary key. Used in PG joins + JSONB refs.| Immutable                | Server, on create |
| `displayName` | What the user typed and reads everywhere. | Free, mutable             | User |
| `name` (slug) | Readable secondary identifier. URL paths, YAML refs, log lines. | Stable; auto-derived from displayName with collision suffix. Optionally editable. | Server, on create |

After the refactor: rename = edit `displayName`. Nothing breaks. References stay valid because they're stored by `id`. The slug is a stable readable handle.

---

## Goals

1. **Rename never breaks references.** Editing `displayName` updates one row, full stop.
2. **YAML stays declarative.** Operators write `secrets: ["openai-prod"]` in a Policy YAML; server resolves slug→id at apply time. They never see ids.
3. **URLs stay readable.** `/control/secrets/openai-prod-2`, not `/control/secrets/01HXY...`.
4. **One source of truth.** PG references use `id`. Slugs in JSON/YAML are surface ergonomics; not stored as cross-refs.
5. **No partial state.** All kinds migrate together in one PR.

## Non-goals

- Soft delete / tombstones. Deleted = gone (matches current behaviour).
- Versioning history. Only the latest spec.
- IDs in YAML. Operators never type or paste an id.
- Renaming the slug independently of displayName. v1 = slug regenerates from displayName on explicit request only (see open question 3).

---

## Design decisions to lock before coding

### 1. ID format

Recommend **`nanoid`** (12 chars, URL-safe alphabet, ~7e21 keyspace). Reasons:
- Shorter than UUID in PG indexes and JSON payloads.
- Looks like an id (`xT8aB2_kZpQR`) — operator never confuses it with a slug.
- Postgres has no native nanoid type; store as `TEXT` with a unique constraint.

UUID is the safe alternative if you want something std-lib-blessed. Pick one.

### 2. URL path: id or slug?

Recommend **slug** for human-readable URLs (`/control/secrets/openai-prod`). API also accepts `id` (`/control/secrets/by-id/{id}`) for stability when slug might change. **Open question.**

### 3. Slug regeneration policy

When `displayName` changes from "OpenAI Prod" to "OpenAI Production", does slug `openai-prod` become `openai-production`?

Recommend **no** — slug is sticky once assigned. Editing slug is an explicit `PATCH` field separate from displayName. Reasons:
- Avoids surprise URL changes.
- Avoids slug churn when users tweak displayName for cosmetics.
- Operators who want a fresh slug ask for it explicitly.

Trade-off: slug can drift from displayName, looking weird (`openai-prod` slug with displayName "Anthropic Production"). Acceptable cost.

### 4. Slug collision handling

On create, if requested displayName slugifies to `openai-prod` and that slug exists, append a suffix: `openai-prod-2`, `openai-prod-3`, etc. The auto-suffix is at server, not client.

Slug regex: `^[a-z0-9](?:[a-z0-9-]{0,62}[a-z0-9])?$` (DNS label rules — same as today's `name`).

### 5. Reference resolution direction

YAML/API write side accepts slug:
```yaml
spec:
  secrets: ["openai-prod"]
```

Server resolves slug→id at apply time, stores as id in JSONB:
```json
{"secrets": ["xT8aB2_kZpQR"]}
```

API read side renders id→slug:
```yaml
spec:
  secrets: ["openai-prod"]
```

The id-form is invisible to YAML/UI consumers. Internal storage is canonical.

**Caveat:** if a referenced resource is deleted, the JSONB has a dangling id. Read-side render shows `<unknown:xT8aB2_kZpQR>` or similar; validation surfaces it.

### 6. Backwards compat

The frontend currently sends `metadata.name` on create. Server treats inbound `metadata.name` as displayName, slugifies it, stamps `id`. Existing rows migrated by:
- Backfill `id` with new nanoid.
- `displayName` = current name.
- Slug = current name (already DNS-valid, by validation).
- JSONB references rewritten in migration: `secrets: ["openai-prod"]` → `secrets: [<id of openai-prod>]`. Migration walks every JSONB column and rewrites in place.

## Surface changes by layer

### PG schema

For every catalog table (`providers`, `policies`, `secrets`, `models`, `routes`, `rate_limits`, `relay_keys`, `passthrough_config`):

```sql
ALTER TABLE providers ADD COLUMN id TEXT NOT NULL DEFAULT gen_nanoid();
ALTER TABLE providers ADD COLUMN display_name TEXT;
-- name column stays as the slug
ALTER TABLE providers DROP CONSTRAINT providers_pkey;
ALTER TABLE providers ADD CONSTRAINT providers_pkey PRIMARY KEY (id);
ALTER TABLE providers ADD CONSTRAINT providers_name_unique UNIQUE (name);
```

`gen_nanoid()` is a SQL function we add (PL/pgSQL or `crypto/random` via a Go-side default — pick one). If Go-side: app stamps `id` before INSERT.

JSONB reference rewrite migration is the trickiest part. Pseudo:
1. Build a `slug → id` map from each table.
2. For each JSONB spec column, parse, rewrite known reference fields, write back.

Reference field map:
- `Policy.spec.secrets[]` → secret ids
- `Policy.spec.models[]` → model ids
- `Policy.spec.rateLimits[].ref` → rate_limit ids
- `Provider.spec.defaultPolicy` → policy id
- `Secret.spec.rateLimits[].ref` → rate_limit ids
- `Model.spec.rateLimits[].ref` → rate_limit ids
- `Route.spec.models[]` → model ids
- `RelayKey.spec.policyRef` → policy id
- `Passthrough.spec.models.allow[]` → model ids

This is exhaustive; new ref fields added later need to extend the map AND extend the rewrite migration if any are added later.

### Go: `internal/catalog`

`Metadata` grows `ID` and `DisplayName`:
```go
type Metadata struct {
    ID          string            `yaml:"id,omitempty" json:"id,omitempty"`           // server-assigned, immutable
    Name        string            `yaml:"name" json:"name"`                            // slug, stable
    DisplayName string            `yaml:"displayName,omitempty" json:"displayName,omitempty"`
    Labels      map[string]string `yaml:"labels,omitempty" json:"labels,omitempty"`
}
```

Drop the per-spec `DisplayName` fields on ProviderSpec/ModelSpec — moved to Metadata.

Reference fields stay typed as `string` but the **values are ids**. A new helper resolves slug→id and id→slug at the boundary.

Snapshot grows two indexes per kind:
```go
providers       map[string]*Provider // keyed by id
providersBySlug map[string]*Provider // keyed by name (slug)
```

Lookup methods:
```go
ProviderByID(id string) (*Provider, bool)
ProviderBySlug(slug string) (*Provider, bool)
// ProviderByName preserved as alias for slug for one release; deprecation warning.
```

### Storage layer

sqlc queries gain `id`. Inserts: app generates id (or PG default). Updates by id. Lists return everything.

Reference resolution is a domain concern, not storage — happens in `internal/catalog/refresolve.go` (new file). Storage stays dumb about what the JSONB strings mean.

### HTTP: admin CRUD

Routes stay `/control/{kind}/{name}` where `{name}` matches a slug. Add a path option `/control/{kind}/by-id/{id}` for stability.

Create: body sends `metadata.displayName` (preferred) or `metadata.name`. Server stamps id, derives slug, returns canonical resource.

Update: full replace, but `id` and slug are server-managed — body's id/name fields are ignored if present, with a warning.

Slug edit: `PATCH /control/{kind}/{name}/slug` body `{to: "new-slug"}`. Validates uniqueness. Optional in v1.

DisplayName edit: regular PUT body sets `metadata.displayName`. Slug unchanged.

### YAML seed

The seed file accepts `metadata.name` (slug) as today. New optional `metadata.displayName`. References use slug as today. Server resolves at apply time. Backwards compatible.

### Frontend impact

OpenAPI regen produces new `Metadata` shape with id/displayName. The UI:
- Lists/cards show `displayName` prominently, slug as monospace secondary.
- Edit forms expose displayName as a regular text field (no validation).
- Slug field is read-only or shown only in an "advanced" panel with explicit edit affordance.
- References (e.g. Policy → Secret picker) display by displayName, send slug on submit.

Frontend changes are mostly mechanical once types regen.

### Validation

- `metadata.id` must match nanoid regex if present (always, after first apply).
- `metadata.name` (slug) must match DNS-1123 regex; unique across kind.
- `metadata.displayName` is free text, max length (256?), no validation otherwise.
- All cross-reference fields validated as ids exist in their target kind's id index.

### Tests

- Snapshot tests with renamed displayName: references still resolve.
- Slug collision: second resource with same slugified displayName gets `-2`.
- YAML seed with slug refs: resolves correctly to id internally.
- Migration test: pre-refactor JSONB with name refs → post-refactor JSONB with id refs.

## Migration plan

1. Add columns + indexes (one migration per table, or a bundle migration `000007_identity.up.sql`).
2. Backfill: id = nanoid, displayName = name, slug = name. Same migration.
3. Rewrite JSONB references: separate migration `000008_id_refs.up.sql`. Walks every JSONB column, parses, rewrites known reference paths, writes back. Idempotent (re-run = no-op).
4. Deploy: apply migrations during normal rollout. Snapshot reload picks up id index.
5. **No downtime requirement** — old snapshot reads work because we preserve name/slug accessors. Pod can run with mixed old/new state during the transition; references are still valid via slug-or-id resolution.

Rollback path: down migrations drop the new columns. References are still slugs in JSONB until the rewrite migration runs, so reverting before that migration is safe. After rewrite migration, rollback is destructive — needs the inverse JSONB rewrite. Document this explicitly: "Once 000008 applies, you can't go back without restoring from backup."
